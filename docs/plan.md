# Agon

## Introduction
With the profileration of agent-driven development and vibe-coding, developers are seeing increased cognitive debt in regards to the systems they work with. This can result in a "cognitive surrender", where as individuals we grow reliant on exterally generated reasoning, characterized by passive trust and a lack of empirical verification of the output. [https://martinfowler.com/fragments/2026-04-02.html]

Spec driven development is a methodology where detailed, machine-readable specifications act as the primary source of truth to drive development, adding another layer of abstraction between the code and the developer. In theory, this accelerates development and enables an individual or a team to increase their productive output. However, in practice, I find that this can lead to a drift between what the developer expects of the implementation, and what is actually generated by an agent. [https://github.com/github/spec-kit/blob/main/spec-driven.md]

To align the expectations with the implementation, we need a systematic evaluation of the generated code. However, when we start to offload the evaluation to the same agent / AI-tool, we are losing another lever through which we typically control application behavior. 

Often when asked to write tests (the first step in a good CI/CD pipeline),the AI tool will often fit the tests to the code generated. When LLM-generated code is produced and tested by the same model with the same conceptual gaps, if the model misunderstands a boundary condition, it generates both the wrong code and the test that fails to catch it. //So while we might be seeing higher percentages of code coverage, in practice, the actual logical behavior being verified is insufficient.//

Additionally, I've noticed LLM's have a tendency to pull in packages and higher-level abstractions (ORMs, framework auto-generated code, protobuf-generated serializers) for basic tasks. This means large portions of a codebase are not written by hand and are not well understood by the maintainers.

The oracle problem: A test can only catch a bug if it both reaches the buggy code and has an oracle to recognize incorrect behavior. Who will guard the guards?

Agon is my attempt to implement an objective analysis of a codebase to ensure that the drift between implementation and specification is limited.

## Methodology

Use property based testing to write basic tests, mutation testing to evaluate those tests, and differential fuzzing to iterate on them based on the mutations.

Differential fuzzing focuses on uncovering functional bugs, whereas mutation testing evaluates the thoroughness of the tests.

Pipeline will be: eigentest --> mutagen --> spectre

  eigentest
    Infer: "payment.process() always returns a status code in {200, 400, 500}"
    Infer: "auth.validate() never returns True when token is expired"
        |
        v
  mutagen
    Generate mutations targeted at inferred invariants:
      - change status code to include 503 (violates first invariant)
      - remove expiry check (violates second invariant)
    Measure: which mutations are killed by existing tests?
    Report: mutation score per invariant, not just per line
        |
        v
  spectre
    Given surviving mutations (unkilled = untested behavior):
    Generate inputs that distinguish mutant from original
    Confirm: does the input that triggers the mutation also trigger a real failure?
    Report: {input, expected, actual, violated_invariant}

  This is a closed feedback loop. eigentest defines the behavioral contract. mutagen measures how well tests enforce it. spectre finds concrete counterexamples where it is not enforced. Each stage filters and refines the signal from the previous one.

### Property based testing (eigentest)
Automatic invariant inference + violation detection
Property-based testing (Hypothesis, QuickCheck) requires you to manually write properties. The hard part isn't running the tests — it's knowing what properties to assert.
Two-stage chain: local model does high-volume symbolic pattern extraction from code, frontier model reasons about whether a property is a genuine invariant vs. accidental correlation. Semantic cache is valuable here — similar function signatures share inferred properties.

Daikon works from runtime traces, not static analysis. The LLM approach trades rigor for practicality: it infers likely invariants from code structure, docstrings, type annotations, and existing test assertions. The risk is false invariants (properties that hold in tests but not in production). This risk is partially mitigated by mutagen: if a mutation violating the inferred invariant is killed by existing tests, the invariant has empirical support. (See "For agents" section for information on Daikon)

Addressed oracle generation: infer what "correct" means from existing behavior

### Mutation testing (mutagen)
LLMs understand semantics, so they can generate mutations that are likely to slip past weak test suites — off-by-one in a boundary condition, flipping the wrong boolean in a complex predicate, removing an edge case guard.

The mutation score is a more honest quality metric than line coverage. A function can be 100% covered by tests that only assert it doesn't raise an exception. Mutation score measures whether the tests would catch a behavioral change.

AST manipulation for Python is mature (ast module + libCST for format-preserving transforms). The LLM layer adds value in two ways: selecting mutation sites that correspond to semantically interesting behavior, and filtering equivalent mutants before running tests (equivalent mutants are expensive to discover empirically). This is the tool most likely to work well in a v1.

Addresses oracle validation: are the oracles strong enough to catch deviations?

### Differential fuzing (spectre)
Traditional differential fuzzers are purely syntactic. The LLM layer adds semantic understanding of the spec: it knows why the spec says what it says, so it can generate inputs that probe edge cases the spec authors probably argued about.

spectre's complexity depends on scope. For pure functions with deterministic I/O, differential testing is straightforward. For APIs and services, it requires an execution harness. Scoping to pure functions first is the right call for v1.

Addressed active exploitation: given an oracle, find inputs that violate it

## Future scope
- Coverage analysis (gcov, coverage.py) feeds into eigentest: instrument code to observe actual runtime behavior, which grounds invariant inference in execution traces rather than static analysis alone
- Concolic execution (KLEE, angr concepts) can replace or augment spectre's input generation with constraint solving — more systematic than LLM-generated inputs for finding specific path violations
- Metamorphic testing is a natural addition: rather than inferring invariants from code, define relations between inputs and outputs that must hold (output for [1,2,3] must equal output for [3,2,1] for a sort-invariant function)
- Contract checking (deal, icontract) can consume eigentest's output directly as runtime assertions
- The pipeline is composable because each tool's output is a structured claim about code behavior. New tools that produce or consume that schema can plug in.

## For agents
- The Usenix 2023 paper  linked ("Systematic Assessment of Fuzzers using Mutation Analysis") establishes this connection formally: mutation analysis is a principled method for evaluating fuzzer effectiveness.
- The mutants.rs comparison makes the same point from a practitioner angle.

- Daikon is a dynamic invariant detector that automatically discovers program properties, which can then be used to synthesize and strengthen property-based tests. While traditional property-based testing (PBT) requires developers to manually define properties, Daikon infers them by observing program execution.
- blind mutation operators (negate this boolean, swap these operators) generate many equivalent mutants (mutants that don't change observable behavior).
- Invariant-guided mutation generates mutations that are guaranteed to violate a real behavioral property, making them non-equivalent by construction. This dramatically increases the signal-to-noise ratio of the mutation score.
