# Agon — Technical Analysis

## Executive Assessment

Agon addresses a real and growing problem: as AI-generated code proliferates, the gap between what developers *think* their code does and what it *actually* does is widening. The three-stage pipeline (eigentest → mutagen → spectre) is architecturally sound — each stage produces structured output that feeds the next, and the closed feedback loop is the strongest design decision in the plan. This is a project worth building.

That said, the plan has significant gaps in feasibility analysis, scoping, and several areas where the ML/LLM integration strategy needs refinement. Below is a systematic breakdown.

---

## 1. Thesis Evaluation

### What's Strong

The framing around "cognitive surrender" and the oracle problem is precise and well-supported. The observation that LLM-generated tests are fitted to LLM-generated code — sharing the same conceptual blind spots — is the strongest insight in the document. This is empirically observable and not adequately addressed by existing tooling.

The Martin Fowler reference and the spec-driven development critique establish credible grounding. The plan correctly identifies that higher code coverage percentages are misleading when the tests themselves are generated by the same model that wrote the code.

### What's Missing

**The adversarial framing is incomplete.** The plan assumes the LLM used in Agon itself won't share the same blind spots as the LLM that generated the code. But if you use GPT-4o to infer invariants on code that GPT-4o wrote, you risk the same conceptual correlation. The plan mentions a two-stage chain (local model + frontier model) but doesn't address *model diversity* as a first-class concern. This is arguably the most critical architectural decision Agon needs to make.

**No discussion of false positive economics.** Any tool that reports "your tests are insufficient" needs to have a very low false-positive rate to remain useful. Developers will ignore a tool that cries wolf. The plan doesn't discuss precision/recall tradeoffs or how to tune the signal-to-noise ratio at each stage.

**The "who guards the guards" question is posed but not fully answered.** Agon itself is an oracle. What validates Agon's inferred invariants? The plan partially addresses this (mutagen provides empirical support for eigentest's inferences), but the bootstrap problem remains: if the initial invariant inference is wrong, the entire pipeline produces confident but incorrect results.

## 2. Pipeline Architecture

### eigentest (Invariant Inference)

**Strengths:**
- The two-stage chain (local model for pattern extraction, frontier model for reasoning) is the right separation of concerns
- Leveraging docstrings, type annotations, and existing test assertions as signal sources is practical
- The acknowledgment that this trades rigor for practicality (vs. Daikon) is honest

**Shortcomings:**

1. **Invariant taxonomy is undefined.** The plan gives two examples (`status code ∈ {200, 400, 500}` and `validate() never returns True when expired`), but doesn't categorize what *kinds* of invariants eigentest should infer. Without a taxonomy, the implementation will ad-hoc its way into inconsistency. At minimum, distinguish:
   - Value range invariants (return values, parameter bounds)
   - Relational invariants (input-output relationships)
   - Temporal/ordering invariants (call sequences, state transitions)
   - Nullability/existence invariants
   - Idempotency and commutativity properties

2. **Static vs. dynamic analysis tradeoff is underspecified.** The plan says the LLM approach "infers likely invariants from code structure" (static) but the future scope mentions runtime traces. For v1, is this purely static? If so, the invariant quality will be substantially lower for code with complex control flow, callbacks, or dynamic dispatch. The plan should commit to a position and explain the tradeoff.

3. **Semantic cache strategy is mentioned but not designed.** "Similar function signatures share inferred properties" — this is a good intuition, but signature similarity is a weak proxy for behavioral similarity. Two functions with signature `(str, int) -> bool` can have completely unrelated invariants. The cache key needs to incorporate structural features (AST shape, dependency graph position, docstring embeddings) not just type signatures. More on this in the ML section.

4. **Scale concern.** A nontrivial codebase has thousands of functions. Inferring invariants for each one with a frontier model is expensive. The plan needs a prioritization strategy — which functions matter most? Heuristics: functions with high cyclomatic complexity, functions touched in recent commits, functions with low existing test coverage, public API surfaces.

### mutagen (Mutation Testing)

**Strengths:**
- This is correctly identified as the most viable v1 component
- AST manipulation via `ast` + `libCST` is a mature, well-understood approach
- LLM-guided mutation site selection is a genuine improvement over blind operators
- Invariant-guided mutation (non-equivalent by construction) is the key insight that makes this tractable

**Shortcomings:**

1. **Equivalent mutant detection is listed as an LLM responsibility, but this is an unsolved problem.** The plan says the LLM filters equivalent mutants "before running tests," but detecting equivalent mutants is undecidable in the general case. The LLM will be wrong sometimes. The plan needs a fallback: run the test suite on suspected-equivalent mutants with a timeout, and track the LLM's accuracy over time to calibrate confidence thresholds.

2. **Mutation execution strategy is unaddressed.** Mutation testing is computationally expensive — each mutant requires a full (or partial) test suite run. The plan doesn't discuss:
   - Test selection: which tests are relevant to a given mutation?
   - Parallelization strategy
   - Incremental mutation (only re-test changed code paths)
   - Timeout handling for mutations that cause infinite loops

   These are engineering concerns, not research concerns, but they determine whether the tool is usable on real codebases.

3. **The mutation score "per invariant" framing is novel but needs formalization.** Traditional mutation scores are per-line or per-function. Mapping mutations to invariants requires a clear association model. How does mutagen know which invariant a given mutation violates? If eigentest's output includes the code region each invariant covers, this is tractable. If not, the mapping becomes fuzzy.

4. **Language scope.** The plan mentions Python AST tooling specifically. Is Agon Python-only for v1? This should be stated explicitly. If multi-language is a goal, the AST manipulation layer needs an abstraction — tree-sitter is the obvious choice for language-agnostic AST access.

### spectre (Differential Fuzzing)

**Strengths:**
- Scoping to pure functions first is the correct pragmatic decision
- The output format `{input, expected, actual, violated_invariant}` is well-structured and actionable

**Shortcomings:**

1. **"Differential" against what?** Classical differential fuzzing compares two implementations of the same spec. Here, spectre compares the original code against mutants. This is more accurately described as *mutation-guided input generation* than differential fuzzing. The terminology matters because it sets expectations — developers familiar with differential fuzzing will expect multi-implementation comparison.

2. **Input generation for non-trivial types is hard.** The plan doesn't address how spectre generates inputs for:
   - Complex object graphs (a `User` with nested `Address` with nested `Country`)
   - Stateful APIs (inputs depend on prior state)
   - Functions with side effects (database writes, network calls)
   - Functions requiring specific environmental setup

   For v1 (pure functions), this is manageable with type-driven generation (similar to Hypothesis strategies). But the plan should acknowledge this boundary explicitly.

3. **Confirmation step is underspecified.** "Confirm: does the input that triggers the mutation also trigger a real failure?" — what constitutes a "real failure"? If the original code returns a different value than the mutant, that's a detected mutation. But does it mean the original code is *correct*? Only if the invariant is correct. This circularity should be acknowledged.

## 3. ML/LLM Integration Analysis

### Where LLMs Add Clear Value

| Component | LLM Role | Why It Works |
|-----------|----------|--------------|
| eigentest — invariant inference | Reasoning about code semantics, docstrings, naming conventions | LLMs excel at inferring *intent* from natural language artifacts in code |
| mutagen — mutation site selection | Identifying semantically meaningful mutation targets | LLMs understand that `>=` vs `>` in a boundary check is more interesting than swapping variable names |
| mutagen — equivalent mutant filtering | Reasoning about whether a code change is observable | This is a semantic reasoning task that benefits from language understanding |
| spectre — edge case input generation | Generating inputs that probe boundary conditions | LLMs can reason about "what inputs would exercise this code path" |

### Where LLMs Are Risky or Insufficient

1. **eigentest invariant inference as the sole oracle source.** LLM-inferred invariants are probabilistic, not sound. The plan partially mitigates this with mutagen's empirical validation, but the pipeline's correctness is upper-bounded by eigentest's accuracy. **Recommendation:** Supplement LLM inference with lightweight static analysis (type narrowing, null checks, assertion extraction) as a ground truth baseline. The LLM adds invariants *on top of* what can be mechanically derived.

2. **Over-reliance on LLMs for tasks with deterministic alternatives.** AST mutation operators can be enumerated mechanically. The LLM should *select and prioritize* among mechanically generated mutations, not *generate* mutations free-form. Free-form LLM-generated mutations risk syntactically invalid code, non-deterministic output, and hallucinated transformations.

3. **LLM cost at scale.** A codebase with 500 functions, each requiring invariant inference, mutation generation, and input generation, will incur significant API costs. The plan mentions a local model for "high-volume symbolic pattern extraction" — this is the right instinct, but needs to be the default path, not the exception.

### Recommended ML/AI Stack

#### Models

| Layer | Model Type | Purpose | Examples |
|-------|-----------|---------|----------|
| Pattern extraction | Small local model (1-8B) | AST analysis, type extraction, simple invariant templates | CodeLlama 7B, DeepSeek-Coder 6.7B, Qwen2.5-Coder |
| Semantic reasoning | Frontier model | Complex invariant inference, equivalent mutant detection | Claude Sonnet/Opus, GPT-4o |
| Embedding | Code embedding model | Semantic cache, function similarity | Voyage Code 3, OpenAI text-embedding-3-large, Jina Code v2 |

#### Semantic Cache Design

The plan mentions semantic cache but doesn't design it. Here's a concrete approach:

```
Cache Key: embedding(function_ast_skeleton + type_signature + docstring)
Cache Value: list of inferred invariants with confidence scores
Similarity Threshold: cosine similarity > 0.85

Invalidation: on function modification (git diff integration)
Warm-up: batch-process on first run, incremental thereafter
```

**Critical detail:** The cache should store invariants at multiple confidence tiers. High-confidence invariants (mechanically derived from type annotations and assertions) are cached aggressively. LLM-inferred invariants are cached with lower TTL and re-evaluated when the function or its dependencies change.

#### NLP Components

1. **Docstring/comment extraction and normalization** — not a model task; use tree-sitter + regex. Don't over-engineer this.
2. **Spec document parsing** — if Agon consumes spec documents (not just code), use an LLM for structured extraction of requirements into testable claims. This is a high-value integration point not currently in the plan.
3. **Natural language invariant → executable property translation** — the LLM infers "this function should never return negative values" and this needs to become `assert result >= 0`. This translation step needs a well-defined intermediate representation.

#### Additional ML Tools

- **Tree-sitter** — language-agnostic AST parsing. Essential for multi-language support. Prefer over Python's `ast` module even for Python-only v1, as it establishes the right abstraction boundary.
- **Vector database** (for semantic cache) — Qdrant, ChromaDB, or LanceDB for local-first operation. Avoid cloud-dependent solutions for a developer tool.
- **Static analysis integration** — Semgrep rules can provide mechanically-derived invariants as ground truth anchors for eigentest.

## 4. Core Shortcomings

### 4.1 No Specification Input Channel

The plan discusses spec-driven development and the drift between spec and implementation, but Agon doesn't actually *consume* specifications. It infers invariants from code alone. This means Agon can tell you "your tests don't catch this mutation" but can't tell you "your code doesn't match your spec."

**Recommendation:** Add an optional spec input to eigentest. If a spec document (markdown, OpenAPI, protobuf definition, etc.) is provided, eigentest should derive invariants from the spec *and* the code, then flag divergences. This closes the loop on the original thesis.

### 4.2 No Incremental/CI Mode

The plan describes a batch analysis pipeline but doesn't address incremental operation. Developers won't run full-codebase mutation testing on every commit. Agon needs:
- A diff-aware mode that only analyzes changed functions and their dependents
- CI integration that reports results as PR comments or check annotations
- Baseline storage so mutation scores can be tracked over time (regression detection)

### 4.3 No Prioritization or Triage Framework

When Agon finds 200 surviving mutants across a codebase, how does the developer know which ones matter? The plan needs a severity/priority model:
- Mutations in security-sensitive code (auth, crypto, input validation) are critical
- Mutations in hot paths (high call frequency) are high priority
- Mutations in utility functions with many dependents are medium priority
- Mutations in rarely-executed code paths are low priority

This prioritization can itself be LLM-assisted (classify function criticality from context).

### 4.4 Language and Ecosystem Scope is Ambiguous

Python tooling is mentioned explicitly (ast, libCST, Hypothesis). The plan should state clearly: is Agon Python-first? If so, acknowledge the limitations (dynamic typing makes invariant inference harder but also more valuable). If multi-language, describe the abstraction layer.

### 4.5 Execution Sandboxing

Mutation testing and fuzzing execute potentially dangerous code mutations. A mutation could:
- Delete files
- Make network requests
- Corrupt state
- Enter infinite loops

The plan doesn't address sandboxing. For v1, this can be process isolation + timeouts + filesystem snapshots. For production use, container-based isolation (nsjail, bubblewrap, or lightweight containers) is necessary.

### 4.6 The Bootstrap Problem

Agon is most valuable for codebases with insufficient tests — but those are also the codebases where eigentest has the least signal. If there are no tests, no docstrings, and minimal type annotations, eigentest's invariant inference degrades to guessing from variable names and control flow. The plan should address this cold-start scenario explicitly, potentially by offering a "baseline generation" mode that creates an initial test suite before running the full pipeline.

### 4.7 Feedback Loop Termination

The pipeline is described as a "closed feedback loop" but there's no termination condition. When does the loop stop? Options:
- Fixed iteration count
- Mutation score threshold (e.g., stop when >90% of invariant-targeted mutations are killed)
- Diminishing returns (new iterations find fewer surviving mutants)
- Budget-based (token/compute cost limit)

Without a defined termination strategy, the tool either runs forever or stops arbitrarily.

## 5. Architectural Recommendations

### 5.1 Define the Intermediate Representation

The plan's greatest strength is composability — each stage's output feeds the next. But the schema for these intermediate artifacts isn't defined. This is the single most important design decision for v1. Propose:

```
Invariant {
  id: string
  function_ref: {file, name, line_range}
  property: string (human-readable)
  property_code: string (executable assertion)
  confidence: float (0-1)
  source: "static" | "llm_inferred" | "spec_derived" | "runtime_observed"
  evidence: list[string] (what supports this invariant)
}

Mutation {
  id: string
  target_invariant: invariant_id
  original_code: string
  mutated_code: string
  operator: string (what was changed)
  status: "killed" | "survived" | "equivalent" | "timeout"
  killing_test: string? (if killed)
}

Counterexample {
  id: string
  mutation_id: string
  invariant_id: string
  input: json
  expected_output: json
  actual_output: json
  reproducer_code: string
}
```

### 5.2 Start with mutagen

Despite eigentest being first in the pipeline, **mutagen is the most tractable v1 component** (as the plan itself notes). Build mutagen first, hardcode some obvious invariants, prove the mutation testing value proposition, then layer eigentest's inference on top.

### 5.3 Model Diversity as a Design Principle

Use *different* models for eigentest (invariant inference) and spectre (input generation). If the same model wrote the code, inferred the invariants, and generated the test inputs, you've recreated the correlated-blind-spots problem Agon exists to solve. This should be a configurable but opinionated default.

## 6. Competitive Landscape

The plan doesn't discuss existing tools. For completeness:

| Tool | Overlap | Agon's Differentiator |
|------|---------|----------------------|
| mutmut / cosmic-ray | Python mutation testing | Agon's mutations are invariant-guided, not blind |
| Hypothesis | Property-based testing | Agon infers properties automatically |
| Daikon | Dynamic invariant detection | Agon works from static analysis + LLM, no instrumented runs needed for v1 |
| Meta's LLM mutation testing | LLM-guided mutation | Agon's pipeline connects mutation to invariants and fuzzing |
| Diffblue Cover | AI test generation | Java-focused, doesn't do mutation testing |
| CodiumAI / Qodo | AI test generation | Generates tests but doesn't evaluate test *quality* |

Agon's differentiator is the closed loop: infer → measure → exploit → refine. No existing tool connects these three stages.

---

## 7. Verdict

**Build it.** The thesis is sound, the pipeline architecture is well-reasoned, and the problem is real and worsening. The key risks are:

1. **eigentest accuracy** — mitigate by supplementing LLM inference with static analysis and by starting with mutagen
2. **computational cost** — mitigate with semantic caching, incremental analysis, and local models for high-volume tasks
3. **false positive rate** — mitigate with confidence scoring and a prioritization framework
4. **correlated blind spots** — mitigate with model diversity

The strongest path to v1: build mutagen with basic AST mutation operators + LLM-guided site selection for Python. Add eigentest's invariant inference. Then add spectre for surviving mutations. Ship each stage as independently useful, with the full pipeline as the premium experience.
